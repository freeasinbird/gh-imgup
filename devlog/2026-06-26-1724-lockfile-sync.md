# Sync package-lock.json (fixing the #39 self-merge fallout)

Branch `fix/lockfile-sync`. Cleanup after self-merging #39 (the 0.1.1 bump)
before its Codex review landed — which flagged a real P2 I rolled past.

## What was wrong

- The 0.1.1 bump committed `package.json` + `CHANGELOG` but **left
  `package-lock.json` out of the commit** (only those two were `git add`ed), so
  `main` had `package.json` at 0.1.1 and the lockfile at 0.1.0.
- Separately, the lockfile `name` was still the pre-scope `gh-imgup` (drift from
  the #32 scope change to `@freeasinbird/gh-imgup` that never updated the lock).

The published 0.1.1 tarball is unaffected (the lockfile isn't published), but the
repo's two npm-metadata files disagreed — bad for reproducible installs / review.

## Fix

Surgically set the lockfile root + `packages[""]` `name` →
`@freeasinbird/gh-imgup` and `version` → `0.1.1`. Minimal diff (4 fields, no
devDep churn); `npm ci --ignore-scripts` now passes, confirming lock ⇆
package.json sync.

## Lesson (the real one)

Self-merging #39 meant the automated review never gated it — a genuine P2 shipped
to `main` (and to npm). Don't merge PRs here; open and stop for the human, even
when a follow-up is delegated. Also: when bumping a version, stage the lockfile
too (`npm version` updates it — `git add -A` or include it explicitly).
Recorded in memory ([[dont-self-merge-prs]]) and promoted to AGENTS.md as a
Releases gotcha (a version bump must commit the lockfile too).

## Verification

- `git diff` = the 4 name/version fields only. `npm ci` OK. Addresses the Codex
  P2 on #39 (follow-up, since #39 is already merged).
